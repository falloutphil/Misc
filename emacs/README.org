#+title: Emacs Build Recipes

This repository contains separate Emacs build recipes for different platform and display-stack combinations.

Do not mix them.

* Wayland build to use here

For the current Debian 13 / Windows WSL / WSLg / Wayland / PGTK / xwidgets setup, the source of truth is:

- =debian_13_trixie_wayland_pgtk/README.md=
- =debian_13_trixie_wayland_pgtk/emacs-wayland-pgtk-webgtk-build-and-package-trixie.sh=
- =debian_13_trixie_wayland_pgtk/emacs-wayland-pgtk-webgtk-client-install-trixie.sh=
- =debian_13_trixie_wayland_pgtk/emacs_prompt.zsh=

This is the correct family for the current Debian 13 WSLg target:

- Emacs 30.2
- PGTK / Wayland
- xwidgets enabled
- private ICU pinned under =/opt/icu75=
- private WebKitGTK pinned under =/opt/wk240=
- GNU Stow install under =/usr/local/stow/emacs-30.2-pgtkwk=

It borrows the WebKit/ICU pinning and packaging shape from the Trixie X11 recipe,
but keeps the Wayland/WSLg runtime behaviour from the older Bookworm recipe.

Do not switch this build to the X11 recipe.

* Older Debian 12 Wayland build

For the older Debian 12 / WSLg / Wayland / PGTK / xwidgets setup, the source of truth is:

- =debian_12_bookworm_wayland_pgtk/readme.md=
- =debian_12_bookworm_wayland_pgtk/build-emacs-30.2-with-pinned-webgtk.sh=
- =debian_12_bookworm_wayland_pgtk/emacs_prompt.zsh=

This is the correct family for the older Debian 12 binary:

- Emacs 30.2
- PGTK / Wayland
- xwidgets enabled
- private WebKitGTK pinned under =/opt/wk240=
- GNU Stow install under =/usr/local/stow/emacs-30.2-wk=

Do not use this older tree for the Debian 13 WSLg target unless you are intentionally
reproducing the old Bookworm build.

* Active Debian 13 install layout

- WebKitGTK runtime prefix: =/opt/wk240=
- ICU runtime prefix: =/opt/icu75=
- Stowed Emacs package: =/usr/local/stow/emacs-30.2-pgtkwk=
- Public launcher: =/usr/local/bin/emacs=

The build uses =/opt/wk240= specifically so xwidgets work with the pinned WebKit
combination for this WSLg/Wayland setup.

Do not casually change:

- =WK_PREFIX=
- =RUNPATH=
- Wayland / PGTK configure flags
- xwidget runtime wrapper environment

* Build notes

The Wayland build script now assumes ALSA development headers are present:

- =libasound2-dev=

This matters because native Emacs sound on Linux is using ALSA in this build path. Without that package, configure can succeed overall while silently leaving =HAVE_ALSA= unset, which breaks =play-sound-file= even if WSLg audio itself works.

The script now checks for that explicitly.

On WSL/WSLg-style systems, build-time ALSA support is still not sufficient by itself. Native Emacs sound also needs the ALSA-to-Pulse bridge at runtime:

- =libasound2-plugins=
- =alsa-utils=
- =~/.asoundrc= routing ALSA default output to Pulse

Without that runtime bridge, =play-sound-file= can still fail with no usable sound device even after a correct rebuild.
